# 0.1 prerelease Overlay Handle migration

`@lyrd/core@0.1.0-next.3`는 identity 기반 갱신 이름을 명확히 하고, `open()`과
`openOrUpdate()`가 해당 세션을 직접 제어할 수 있는 awaitable Handle을 반환한다.

## Identity API 이름 변경

```diff
- overlay.upsert(definition, identity, input, options)
+ overlay.openOrUpdate(definition, identity, input, options)
```

`openOrUpdate()`는 같은 definition과 identity의 활성 세션이 있으면 input을 갱신하고 기존
Handle을 반환한다. 활성 세션이 없으면 새 세션을 생성한다.

## Awaitable Handle

기존의 단순 호출은 바뀌지 않는다.

```ts
const outcome = await overlay.open(projectSettings, input)
```

반환값을 보관하면 같은 활성 세션을 직접 제어할 수 있다.

```ts
const settings = overlay.open(projectSettings, input)

settings.update(nextInput)
settings.dismiss('programmatic')

const outcome = await settings
```

Handle 자체가 실제 Promise이므로 별도의 `.outcome` property를 기다릴 필요가 없다.

## 역할 구분

- `open()`은 항상 새 세션을 만든다.
- `handle.update(input)`은 Handle이 가리키는 활성 세션만 갱신한다.
- `handle.dismiss(reason)`은 그 세션을 명시적으로 종료한다.
- `handle.dismiss()`는 명시적 Application action이므로 `dismissPolicy`를 우회한다. ESC나
  outside click은 Renderer의 `requestDismiss()`를 사용한다.
- 종료된 Handle의 `update()`와 `dismiss()`는 `false`를 반환하고 새 세션을 만들지 않는다.
- `openOrUpdate()`는 definition과 identity로 활성 세션을 찾고, 없으면 새 세션을 만든다.

한 로컬 흐름이 세션을 소유하면 Handle을 보관한다. WebSocket이나 background event처럼
여러 호출부가 같은 작업을 찾아야 하면 안정적인 업무 identity와 `openOrUpdate()`를 사용한다.

Handle을 제어해야 한다면 `.then()`을 호출한 반환값이 아니라 `open()`의 직접 반환값을
보관한다. Promise의 `.then()`은 일반 Promise를 반환한다.

## Progress 예시

```diff
const upload = overlay.openOrUpdate(uploadProgress, uploadId, initialInput)

- overlay.upsert(uploadProgress, uploadId, nextInput)
+ upload.update(nextInput)

const outcome = await upload
```

다른 호출부가 Handle을 갖고 있지 않다면 같은 identity로 다시 호출할 수 있다.

```ts
overlay.openOrUpdate(uploadProgress, uploadId, nextInput)
```
